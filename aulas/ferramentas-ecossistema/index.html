Alternativas

Bloco 3

🗄️ Onde os dados ficam?

Se o projeto precisa guardar dados que sobrevivem ao refresh, vocês precisam escolher: localStorage ou backend de verdade.

Traga exemplos: cadastro, posts, pedidos, votos, favoritos, upload de arquivo.

BaaS

Backend sem
programar backend

Banco

Tabelas ou documentos prontos.

API

Leitura e escrita automáticas.

Extras

Auth, storage e tempo real.

BaaS = Backend-as-a-Service. Você não escreve rotas; usa um serviço que já entrega banco e API.

Duas opções

Supabase ou Firebase

SupabaseFirebase
Quem éOpen source, PostgreSQLGoogle
BancoSQL, tabelasNoSQL, documentos
CurvaMédiaBaixa
Brilha emDados relacionaisApps simples e tempo real

Recomendação: Supabase para aprender algo útil fora do hackathon; Firebase para entregar rápido se o app for simples.

Como integrar

Visão simplificada

  1. Criar conta e projeto.
  2. Pegar URL + chave pública.
  3. Mandar para a IA com a estrutura desejada.
  4. Colar o código gerado e testar.

Importante: diferencie chave pública de segredo. Mesmo assim, evitar commitar dados sensíveis.

Prompt exemplo

Supabase no app

Integre Supabase no meu app. URL: [minha URL] Anon key: [minha anon key] Crie uma tabela pedidos com campos: id, nome, valor, criado_em. Implemente funções para: - listar pedidos - criar pedido - atualizar a tela após salvar

Não coloque chaves reais em slide ou print público. Use valores fake na apresentação.

Quando NÃO precisa

Não complique

localStorage resolve

  • Calculadora.
  • Conversor.
  • Joguinho local.
  • Landing page.

Backend entra quando

Existe cadastro, dado compartilhado, upload, votos, pedidos ou histórico entre dispositivos.

Essa decisão economiza muito tempo. Não use banco só porque parece mais “profissional”.

Bloco 4

🔐 Login de usuário

Implementar login do zero é um buraco negro de tempo. No hackathon, use serviço pronto — ou pule login.

Diga com firmeza: login raramente é a feature que ganha hackathon. Se não for essencial, pule.

Bloco 6

🐙 GitHub
mínimo viável

GitHub é backup, colaboração e ponte para deploy automático.

Explique para não técnicos: GitHub é a pasta do projeto na nuvem. Commit é um save. Push envia para a nuvem.

3 frases

Git sem trauma

Repo

Pasta do projeto na nuvem.

Commit

Um save com mensagem do que mudou.

Push

Enviar os commits para a nuvem.

Não mencione rebase, branch, merge conflict. Para hackathon iniciante: commit + push resolve 95%.

Fluxo mínimo

GitHub Desktop

  1. Criar conta no GitHub.
  2. Instalar GitHub Desktop.
  3. Criar repo novo.
  4. Apontar para a pasta do projeto.
  5. A cada feature: commit + push.

Mostre que não precisa de terminal. GitHub Desktop é suficiente para o público desta aula.

Por que importa?

GitHub salva hackathon

Deploy

Vercel/Netlify atualizam a cada push.

Backup

Se o PC morrer, o projeto continua.

Visibilidade

O juiz pode pedir o repo organizado.

Recomendação: começar privado e decidir publicar no final.

Cuidado

Não commite
secrets

Nunca

Chaves de API, tokens, senhas e `.env` dentro do GitHub.

Faça

Use `.env`, adicione `.env` ao `.gitignore` e peça para a IA extrair chaves do código.

Esse alerta é essencial. Se a IA gerar chave hardcoded, peça para mover para variável de ambiente.

Stack recomendada

Resposta default segura

Editor: VS Code + Live Server IA texto: AI Studio IA visual: Google Stitch + AI Studio Frontend: HTML + Tailwind + JS vanilla Backend: Supabase, só se precisar Deploy: Netlify ou Vercel Versionamento: GitHub Desktop + repo privado Chat: Discord

Use esse slide quando alguém perguntar “mas qual eu uso?”. Essa stack é simples, gratuita e boa para hackathon.

Demos sugeridas

  1. Supabase criando tabela e persistindo dado.

Não faça demo de AWS/Azure/GCP, nem Git avançado. A aula precisa continuar acessível.
