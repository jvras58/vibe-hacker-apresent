Código precisa de um lugar para rodar, dados para demonstrar e uma história que pessoas consigam experimentar.

Esta aula fecha o percurso. Priorize decisões que desbloqueiam a entrega; listas de ferramentas ficam em recursos.

Quatro decisões

Onde editar, guardar, publicar e mostrar?

Peça para as equipes responderem rapidamente o que já está decidido.

Stack mínima

Escolha o caminho com menos dependências

Editar

VS Code, Cursor ou editor web.

Versionar

GitHub e commits pequenos.

Publicar

Netlify, Vercel ou Pages.

Dados

Local primeiro; backend se necessário.

Os nomes são exemplos atuais, não compromissos permanentes. A decisão deve considerar a experiência da equipe.

Ambiente local

Três ferramentas para enxergar o projeto

Servidor local

Evita limitações de abrir HTML diretamente.

Console

Mostra erros e logs reproduzíveis.

Network

Mostra APIs, falhas e tempo de resposta.

Faça uma demo rápida do DevTools se o grupo ainda não o conhece.

Publicação

Deploy cedo, não cinco minutos antes

Publique uma versão mínima e abra o link em outro dispositivo.

Critério

Uma pessoa fora do computador do time consegue acessar e completar a jornada principal.

Deploy cedo reduz surpresa de caminhos, arquivos, APIs e permissões.

Dados

Comece com a solução mais simples que atende a demo

Local ou fictício

Landing, calculadora, catálogo demonstrativo e protótipo navegável.

Backend

Dados compartilhados, votos, uploads, histórico ou uso entre dispositivos.

Não adicione banco para parecer profissional. Adicione quando o comportamento exige persistência compartilhada.

Login

Cadastro é uma feature, não uma decoração

  • A jornada principal exige identidade?
  • A pessoa perde algo importante sem conta?
  • O login melhora a demo ou cria atrito?
Se a resposta for “não”, pule o login.

Para projetos LGBTQIAPN+, cadastro pode aumentar risco e excluir pessoas. Justifique a coleta.

Versionamento

Git é o botão de voltar do projeto

Commit

Uma mudança coerente e testada.

Push

Backup e colaboração.

Revert

Retorno a um estado funcional.

GitHub Desktop é aceitável. O objetivo pedagógico é preservar estados funcionais.

Segredos

Chave privada nunca entra no repositório ou no slide

Nunca

Token, senha, chave privada, `.env` ou dado real em print público.

Faça

Variáveis de ambiente, `.gitignore`, valores fictícios e rotação após vazamento.

Diferencie chaves públicas de clientes e segredos de servidor, mas mantenha a regra simples para iniciantes.

Time

Produto

Escopo e decisão.

Experiência

Design e conteúdo.

Entrega

Código e deploy.

Qualidade

Teste e demo.

As pessoas podem acumular papéis, mas cada responsabilidade precisa ter alguém olhando.

Checklist comunitário

Antes de publicar para pessoas reais

  • Dados sensíveis foram removidos?
  • Localização e identidade estão protegidas?
  • Conteúdo público possui canal de denúncia?
  • O fluxo funciona sem revelar mais do que o necessário?
  • Texto, contraste e teclado foram revisados?

Se o projeto não consegue responder, mantenha-o como protótipo com dados fictícios.

Prática • Trace a rota até a demo

Qual é o mínimo necessário para mostrar valor?

8 minutos
  1. Defina a jornada que será mostrada.
  2. Nomeie o estado mínimo funcional.
  3. Escolha dados e publicação.
  4. Defina quem executa o teste final.
  5. Organize a história de 60 segundos.

Folha • Etapa 5

Jornada da demo, estado mínimo, dados, publicação, responsável pelo teste e próximo passo.

Se houver tempo, cada equipe apresenta: foco, direção, comportamento, critério, risco, evidência e próximo passo. Não é necessário ter uma aplicação pronta.
