O problema do hackathon

Cada prompt vira uma ilha

Sábado, 14h

Você pediu login às 10h, dashboard às 12h e relatórios às 14h.

Troca email por usuário, inventa banco diferente e quebra o que já funcionava.

Bloco 1

O que é uma
spec?

Documento curto em Markdown que descreve o que o software faz, por quê, e com quais restrições — antes de uma linha de código existir.

Evite explicar como documentação formal pesada. A spec aqui é curta, prática e feita para o agente de IA usar.

Analogia simples

A reforma da casa

Vibe coding puro

Você manda mensagem pro pedreiro toda hora: “põe tomada ali”, “pinta de azul”, “mudei de ideia”.

Resultado: Frankenstein.

Spec-driven

Você faz a planta, define cores, materiais e prazo. O pedreiro executa a planta.

A spec é a planta da casa.

Essa analogia funciona bem com público não técnico. Deixe claro: mudança pode acontecer, mas passa pela planta antes de virar parede.

Visão geral

O que a spec contém?

  1. As leis do projeto.
  2. O que construir.
  1. Como construir.

2.1 constitution.md

A lei fundamental

O que é?

Regras que valem para o projeto inteiro, do começo ao fim.

O que entra?

  • Stack obrigatória.
  • Convenções de código.
  • Padrões de UX.
  • O que é proibido.

Diga: se a IA toma uma decisão que viola a constitution, isso é bug. Você corrige citando o arquivo.

Exemplo

constitution.md de hackathon

# Constituição do Projeto: GrãoBravo ## Stack - Frontend: HTML + Tailwind CSS + JavaScript vanilla - Backend: nenhum - Persistência: localStorage ## Princípios - Mobile-first: testar em 360px antes de tudo. - Tempo de carregamento < 2s. - Sem dependências pagas. - Sem login. Tudo público. ## Voz da marca - Tom amigável, descontraído, sem termos técnicos. - Português brasileiro. - Emojis com moderação.

Explique que a constitution não precisa ser longa. Cinco a quinze linhas já mudam muito o comportamento do agente.

2.2 spec.md

O que construir

Ponto de vista

Feature descrita do ponto de vista do usuário, sem código e sem detalhe técnico.

Conteúdo

  • Quem é o usuário.
  • Comportamento esperado.
  • Critérios de aceitação.
  • Casos extremos.

Reforce “sem código”. Se a spec já começa com biblioteca e implementação, ela pulou uma etapa.

Exemplo

spec.md: carrinho

# Feature: Carrinho de Compras ## Usuário Cliente navegando os produtos da cafeteria. ## Comportamento 1. Cada card tem botão "+ Carrinho". 2. O badge no header incrementa. 3. O ícone abre painel lateral. 4. O usuário altera quantidade ou remove. 5. O total recalcula em tempo real.
## Critérios de aceitação - [ ] Sobrevive a refresh da página. - [ ] Total em formato BRL. - [ ] Carrinho vazio mostra mensagem. ## Fora de escopo - Cupons de desconto. - Frete. - Pagamento.

Chame atenção para “Fora de escopo”. Sem isso, a IA ajuda demais e implementa coisas não pedidas.

Detalhe importante

Fora de escopo também é spec

Sem limite

A IA “ajuda demais”: adiciona cupom, pagamento, frete, login e quebra o prazo.

Com limite

A IA sabe exatamente o que NÃO deve fazer agora.

Essa é uma das maiores vantagens práticas para hackathon: limitar escopo salva tempo.

2.3 plan.md

Como construir

Quem cria?

Geralmente a própria IA, baseada na constitution + spec. Você revisa.

O que entra?

  • Arquivos.
  • Estrutura de dados.
  • Bibliotecas.
  • Fluxo técnico.

Mostre que plan é a ponte entre linguagem de usuário e código. Ainda é checkpoint humano.

Exemplo

plan.md recortado

# Plano: Carrinho de Compras ## Arquivos - cart.js — lógica do carrinho: add, remove, total. - cart-ui.js — manipulação do DOM: painel e badge. - cart.css — estilos do painel lateral. ## Estrutura de dados Carrinho = array no localStorage: { id, nome, preco, quantidade } ## Bibliotecas Nenhuma, conforme constitution.

Se a IA sugerir biblioteca que viola constitution, pare aqui. Corrigir no plan é barato; corrigir depois no código é caro.

2.4 tasks.md

Passos executáveis

A feature vira tarefas pequenas que a IA executa uma por vez, marcando o que já foi concluído.

# Tarefas: Carrinho de Compras - [ ] 1. Criar cart.js com addItem, removeItem, getTotal. - [ ] 2. Criar cart-ui.js com renderPainel e atualizarBadge. - [ ] 3. Adicionar botão "+ Carrinho" nos cards. - [ ] 4. Implementar painel lateral. - [ ] 5. Conectar clique do ícone ao painel. - [ ] 6. Implementar localStorage. - [ ] 7. Testar carrinho vazio. - [ ] 8. Testar refresh da página.

Diga: com tasks, dá para pausar no meio e voltar amanhã. A IA sabe onde parou.

Bloco 3

O fluxo
passo a passo

/constitution→/specify→/plan→/tasks→/implement

Explique que esses são comandos do Spec Kit em agentes compatíveis. No AI Studio dá para fazer o mesmo fluxo manualmente.

Comandos do Spec Kit

Digite literalmente

/constitution → cria a constitution.md /specify → cria uma spec.md a partir da feature /plan → gera o plan.md a partir da spec /tasks → quebra o plan em tasks.md /implement → executa as tasks, uma de cada vez

Não entre em instalação ainda. Primeiro explique o fluxo mental.

Checkpoints humanos

A palavra mais importante:
APROVA

Spec

A IA propõe. Você revisa.

Plan

A IA planeja. Você valida.

Tasks

A IA quebra. Você aprova.

Esse é o ponto que diferencia SDD de deixar a IA trabalhar solta. O humano guia o trilho.

Por que melhora?

Menos retrabalho

  • Erro aparece cedo.
  • Mudança de ideia custa menos.
  • A IA não inventa fora do trilho.
  • Você pode pausar e voltar depois.

Exemplo: mudar uma spec leva minutos; refazer código depois de 3 horas de agente leva muito mais.

Uso real no hackathon

Linha do tempo

Sexta 19h
constitution → 19h30
spec login → 20h
plan + tasks → Sábado 9h
dashboard

Mostre que as decisões da noite anterior continuam valendo porque estão no arquivo, não no “achismo” do chat.

4.1 GitHub Spec Kit

O molde dos comandos

O que é?

Ferramenta open-source do GitHub que implementa o fluxo dos 5 comandos.

Onde funciona?

GitHub Copilot, Claude Code, Cursor e Gemini CLI.

Explique “open-source, agnóstico de agente, gratuito”. Evite instalar ao vivo se puder falhar.

Instalação rápida

Spec Kit

# Requer uv uvx --from git+https://github.com/github/spec-kit specify init meu-projeto

Depois, abra o editor no projeto. Os comandos aparecem no chat do agente.

Não gaste a aula depurando instalação. Recomende deixar instalado antes ou usar o caminho do AI Studio.

O que ele gera

Estrutura de pasta

meu-projeto/ ├── .specify/ ├── constitution.md ├── specs/ │ └── 001-carrinho/ │ ├── spec.md │ ├── plan.md │ └── tasks.md └── src/

Destaque o histórico versionado: cada feature vira uma pasta numerada com decisões e tarefas.

Atalho do hackathon

GEMINI.md mínimo

Sem Spec Kit?

Crie um arquivo de regras na raiz: GEMINI.md, CLAUDE.md ou .cursorrules.

# Projeto: GrãoBravo Stack: HTML + Tailwind + JS vanilla. Sem build tools. Estilo: dark mode, mobile-first. Idioma: PT-BR no UI, EN no código. Nunca instalar dependências sem perguntar. Sempre fazer commits pequenos.

Esse é o mínimo viável de SDD. Para hackathon, muitas vezes já é suficiente.

4.5 Sem instalar nada

SDD direto
no AI Studio

Você roda o fluxo na mão: System Instructions como constitution, mensagem inicial como spec, plano e tarefas na conversa.

Esse é o caminho mais importante para público não técnico. Eles podem aplicar sem terminal.

Mapeamento direto

Spec Kit → AI Studio

Faça a ligação com a Aula 1: System Instructions é uma constitution persistente da conversa.

Receita prática

System Instructions

Você é parte do time GrãoBravo. Stack: HTML + Tailwind + JS vanilla, sem build tools. Estilo: mobile-first, dark mode, fonte Inter. Idioma: PT-BR no UI, EN no código. Regras de trabalho: - Antes de qualquer código, devolva um PLANO em bullets e espere meu OK. - Quebre features em TAREFAS numeradas. Faça uma por vez. - Nunca instale dependência sem perguntar. - Se eu mudar de ideia, atualize a spec antes do código.

Mostre no AI Studio onde fica o painel System Instructions, se possível com print no próximo slide.

Truques do AI Studio

Use a favor do SDD

  • Anexar constitution.md e spec.md.
  • Salvar o chat do projeto.
  • Usar “Get code” depois.
  • Temperatura baixa para código.

Explique temperatura: para código, 0.2–0.4 reduz improviso fora da spec.

Pirâmide de uso

Do simples ao avançado

AI Studio puro→AI Studio + spec→Gemini CLI + GEMINI.md→Spec Kit completo

Recomende começar simples. Migre só quando o projeto pedir mais organização.

Bloco 5

Os 3 níveis
de maturidade

1. Spec-first

Spec curta antes de prompar. Ideal para hackathon.

2. Spec-anchored

Spec e código vivem juntos no repositório.

3. Spec-as-source

A spec é o projeto. Código pode ser descartável.

Diga claramente: para hackathon, nível 1 é o alvo. O nível 3 é empresarial, não precisa agora.

Quando NÃO usar

SDD tem custo

É exagero se...

  • Dura menos de 2 horas.
  • É script de uso único.
  • Projeto cabe em 1 arquivo.

Vai salvar se...

  • Time tem 2+ pessoas.
  • Dura mais de 1 dia.
  • Tem mais de 3 features.

Apresente SDD como ferramenta, não dogma. Bom senso vence.

Bloco 6

Checklist da spec mínima

  • Criei constitution com stack, estilo, restrições e idioma.
  • Coloquei na raiz do projeto.
  • Escrevi spec de 1 página por feature.
  • Incluí usuário, comportamento e aceitação.
  • Marquei o que está fora do escopo.
  • Atualizei a spec antes do código quando mudei de ideia.
